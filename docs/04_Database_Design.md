## 2. Table Definitions

### 2.1 STUDENTS

```sql
CREATE TABLE students (
    id              BIGINT          GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name            VARCHAR(255)    NOT NULL,
    username        VARCHAR(255)    NOT NULL UNIQUE,
    password        VARCHAR(255)    NOT NULL,
    raw_password    VARCHAR(255),
    device_lock     BOOLEAN         DEFAULT FALSE,
    show_recordings BOOLEAN         DEFAULT TRUE,
    blocked         BOOLEAN         DEFAULT FALSE,
    muted           BOOLEAN         DEFAULT FALSE,
    status          VARCHAR(255)    DEFAULT 'OFFLINE',
    created_at      TIMESTAMP,
    last_seen       TIMESTAMP
);
```

**Indexes:**
| Index Name              | Column(s)   | Type   |
|-------------------------|-------------|--------|
| PRIMARY (auto)          | id          | PK     |
| UK_students_username    | username    | UNIQUE |

| 2  | Priya | PRIYA01  | pass1234     | ONLINE  | false   | false |

## 3. Relationship Summary

| Relationship                     | Type              | Enforcement | Notes                                   |
|----------------------------------|-------------------|-------------|------------------------------------------|
| Student → Recording              | One-to-Many       | Logical FK  | recordings.student_id → students.id      |
| Meeting → Recording              | One-to-Many       | Logical FK  | recordings.meeting_id → meetings.id      |
| Meeting → ChatMessage            | One-to-Many       | Logical FK  | chat_messages.meeting_id → meetings.id   |

> **Note:** Foreign keys are logical (application-enforced) rather than database-enforced constraints. 
> This is typical for H2 in-memory databases with `create-drop` DDL strategy where JPA/Hibernate 
> manages the relationships at the application level.

## 4. Data Access Patterns

### 4.1 Query Patterns by Repository

| Repository            | Query Method                                            | SQL Pattern                                      |
|-----------------------|---------------------------------------------------------|--------------------------------------------------|
| StudentRepository     | `findByUsername(username)`                               | `SELECT * FROM students WHERE username = ?`      |
| StudentRepository     | `findByStatus(status)`                                  | `SELECT * FROM students WHERE status = ?`        |
| StudentRepository     | `findByNameContaining...OrUsernameContaining...(k, k)`  | `WHERE LOWER(name) LIKE ? OR LOWER(username) LIKE ?` |
| StudentRepository     | `existsByUsername(username)`                             | `SELECT COUNT(*) > 0 FROM students WHERE username = ?` |
| TeacherRepository     | `findByUsername(username)`                               | `SELECT * FROM teachers WHERE username = ?`      |
| MeetingRepository     | `findByActive(active)`                                  | `SELECT * FROM meetings WHERE active = ?`        |
| ChatMessageRepository | `findByMeetingIdOrderBySentAtAsc(meetingId)`            | `WHERE meeting_id = ? ORDER BY sent_at ASC`      |
| ChatMessageRepository | `deleteByMeetingId(meetingId)`                          | `DELETE FROM chat_messages WHERE meeting_id = ?`  |
| RecordingRepository   | `findByStudentId...OrderBy...Desc(studentId)`           | `WHERE student_id = ? ORDER BY recording_date DESC, recording_time DESC` |
| RecordingRepository   | `findByMeetingId(meetingId)`                            | `WHERE meeting_id = ?`                           |
| RecordingRepository   | `deleteByStudentId(studentId)`                          | `DELETE FROM recordings WHERE student_id = ?`    |
| AuditLogRepository    | `findByActorOrderByTimestampDesc(actor)`                | `WHERE actor = ? ORDER BY timestamp DESC`        |
| AuditLogRepository    | `findByActionOrderByTimestampDesc(action)`              | `WHERE action = ? ORDER BY timestamp DESC`       |
| AuditLogRepository    | `findByTimestampBetween...Desc(from, to)`               | `WHERE timestamp BETWEEN ? AND ? ORDER BY timestamp DESC` |
| AuditLogRepository    | `findTop100ByOrderByTimestampDesc()`                    | `ORDER BY timestamp DESC LIMIT 100`              |

### 4.2 Transaction Configuration

| Service            | Default TX  | Read-Only Methods                              |
|--------------------|-------------|------------------------------------------------|
| MeetingService     | Read-Write  | getActiveMeeting, getAllMeetings, findById      |
| StudentService     | Read-Write  | getAllStudents, findById, search, counts         |

| RecordingService   | Read-Write  | getStudentRecordings, getAllRecordings           |

## 5. Database Configuration

```properties
# H2 In-Memory Database
spring.datasource.url=jdbc:h2:mem:mtngdb;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE
spring.datasource.driverClassName=org.h2.Driver
spring.datasource.username=sa
spring.datasource.password=

# JPA / Hibernate
spring.jpa.hibernate.ddl-auto=create-drop
spring.jpa.show-sql=false
spring.jpa.properties.hibernate.format_sql=true
spring.jpa.defer-datasource-initialization=true
spring.jpa.open-in-view=false

# H2 Console
spring.h2.console.enabled=true
spring.h2.console.path=/h2-console
```

### 5.1 Configuration Notes

| Parameter                              | Value         | Purpose                                           |
|----------------------------------------|---------------|---------------------------------------------------|
| `DB_CLOSE_DELAY=-1`                   | JDBC URL      | Keeps H2 DB alive while JVM runs                  |
| `DB_CLOSE_ON_EXIT=FALSE`              | JDBC URL      | Prevents DB shutdown on last connection close      |
| `ddl-auto=create-drop`                | Hibernate     | Recreates schema on every startup                  |
| `defer-datasource-initialization=true` | JPA           | Ensures DataInitializer runs after schema creation |
| `open-in-view=false`                  | JPA           | Prevents lazy loading in view layer                |

## 6. Data Volume Estimates

| Table          | Expected Rows (per session) | Growth Rate  | Notes                          |
|----------------|----------------------------|--------------|--------------------------------|
| teachers       | 2-5                        | Static       | Seeded at startup              |
| students       | 10-50                      | Low          | Created by admin               |
| meetings       | 1-10 per day               | Moderate     | Each session creates one       |
| chat_messages  | 50-500 per meeting         | High         | Cleared per meeting optionally |
| recordings     | 10-50 per meeting          | Moderate     | Auto-saved on meeting end      |
| audit_logs     | 100-1000 per day           | High         | All events logged              |

> **Note:** Since H2 is in-memory with `create-drop`, all data resets on application restart.
> For production use, migrate to PostgreSQL or MySQL with `update` DDL strategy.

## 7. Migration Path (Future Production)

| Aspect            | Current (H2)                 | Production Recommendation        |
|-------------------|------------------------------|----------------------------------|
| Database          | H2 In-Memory                 | PostgreSQL 16+ / MySQL 8+        |
| DDL Strategy      | `create-drop`                | `validate` with Flyway/Liquibase |
| Connection Pool   | Default HikariCP             | HikariCP (tuned)                 |
| Foreign Keys      | Logical (app-enforced)       | Physical FK constraints           |
| Indexing           | Primary + Unique only       | Add indexes on FK columns, status |
| Persistence       | Lost on restart              | Persisted on disk                 |
| Console Access    | H2 Console (`/h2-console`)  | pgAdmin / MySQL Workbench        |
